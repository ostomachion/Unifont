Unifont Glyph Modifications and Extensions

Emoji Glyph Consolidation

The first goal of this work is to improve the emoji glyphs in GNU Unifont with a focus on three consistency criteria listed below in order of priority:

  1. Be consistent within Unifont. Many emoji glyphs in the font exhibit inconsistencies in shapes, sizes, and positions, both in their overall forms and in their individual components. For instance, while most of the “smiley” emoji are drawn within a standard 15×14 circle, some deviate from this standard size without a clear reason. Similarly, the shapes and positions of commonly used eye and mouth types can be standardized. These adjustments will help create a more cohesive look and feel across the set, while still allowing for individual glyphs to retain their unique character where appropriate.

  2. Be consistent with the official guidelines. The glyph designs should adhere to the specifications and recommendations outlined in the various emoji and CLDR documents and data files. Extensive details can be found in the Design Guidelines section of Unicode® Technical Standard #51.

    Many existing glyphs appear to have been designed solely based on the formal character name and representative glyph in the Unicode code charts, without sufficient consideration for the information provided by the emoji and CLDR documents and data files. For example, the code point U+1F603 has the formal name “SMILING FACE WITH OPEN MOUTH”, but a CLDR short name of “grinning face with big eyes”. As noted in the Unicode documentation:

    “The formal Unicode name of a Unicode character does not determine its appearance. […] In many cases, the consensus for the best depiction has evolved in the time since the original formal name was standardized, and the preferred depiction is now better reflected by the CLDR short name.”

  3. Be consistent with industry practice. This criterion is specifically detailed in Unicode® Technical Standard #51 but is explicitly listed here for emphasis. Many emoji fonts and vendors have agreed upon a set of visual conventions and styles that increase interoperability and user familiarity across platforms. Aligning with these practices helps maintain consistency and avoids user confusion when interpreting emoji designs.

    While consistency with industry practices is encouraged, it is important to recognize that exact replication of existing designs is neither required nor always practical. Reasonable interpretation and creativity in representing emoji and other icons are valid and often necessary to accommodate diverse design contexts or technical constraints. The goal is to balance user expectations with flexibility in artistic expression and functionality.

Extended Emoji Support

The second goal of this work is to create extensions to GNU Unifont to add full support for emoji. These extensions are likely out of scope for the Unifont project, in which case they will continue to be maintained and provided separately on this page. The extensions include:

  • glyphs for the emoji variation sequences: text style and emoji style variation sequences
  • glyphs for the basic emoji sequences: keycap, flag, tag, and RGI modifier sequences
  • glyphs for the emoji RGI ZWJ sequences: family, role, gender, hair, and other RGI ZWJ sequences
  • colored versions of each emoji glyph

Color Palette

The color palette used for the color glyphs is a custom palette that will be called “Human16” in this document.

The Human16 palette was carefully designed with the following goals in mind:

  • Be simple enough to fit with the look, feel, and philosophy of Unifont. Human16 uses only 16 colors (not counting transparency), with no two colors being too similar. An efficient glyph encoding can be derived analogous to the .hex format used for existing monochrome Unifont glyphs, using four bits per pixel instead of one. Either a simple transparency mask can be used, or a simple algorithm can automatically determine the transparent pixels as none of the outermost pixels of the glyph will ever be pure white.
  • Be compatible with all existing emoji characters and their expected presentations. Human16 includes the 12 colors of the explicitly colored shape emoji (hearts, squares, circles, etc), as well as the five skin tones from the set of skin tone modifier emoji components (one of which overlaps with the basic “brown” color).
  • Be accessible. Human16 incorporates the colorblind safe palette used by Bang Wong (original source; see also Wong, B. Points of view: Color blindness. Nat Methods 8, 441 (2011) for more details). This palette is expanded by adding a carefully selected shade of purple and gray.
  • Be inclusive. Rather than the Fitzpatrick scale, Human16 derives its skin tone colors from Monk, Ellis. “Monk Skin Tone Scale,” 2019. This scale is more modern, inclusive, and representative than the Fitzpatrick scale which is primary a dermatological tool and has been criticized for its Eurocentric bias and insufficient representation of global skin color diversity. The specific tones have been modified to better match the the rest of the palette.

The following table presents the proposed glyph designs. The Unifont column shows the current glyph in Unifont 16.0.02. The New columns show the new glyph designs in text style (if the code point has a text style variation), monochrome emoji style, and color emoji style.

If the character has a default text style presentation, the glyph in the Unifont column will be outlined in yellow. If Unifont adopts the new designs in its charts, it will have to decide whether to use the text style or (monochrome) emoji style glyph variant for those characters that have variations. Several possible approaches are listed below.

  • Use the default style. This aligns with many of the glyph designs in Unifont 16.0.02. (See the half-width U+263A as an example.) This may align with user expectations in most cases, although a few edge cases may cause confusion.
  • Always use the text style when one exists. This may be a good choice considering that Unifont does not have any “support” for emoji other than providing monochrome glyphs for the base characters.
  • Always use the (monochrome) emoji style. This brings the most consistency to all emoji glyphs, although the text style may be preferred or expected in some cases.
  • Incorporate both styles by adding support for emoji variation sequences to Unifont. This gives users and systems control over the glyphs at the expense of the simplicity of Unifont.
  • Create separate fonts for text style vs emoji style. There is precedence for this approach with the Japanese versions of the CJK glyphs. A similar approach could be used for emoji at the expense of multiplying the number of charts and fonts.

My possibly unrealistic recommendation would be for Unifont to use the default presentation in the charts, and then design a mechanism to support character sequences. At a minimum, an initial implementation could add support only for variation sequences (in order to support both text style and emoji style presentations). Future implementations could then support the other emoji sequences if desired. However, I understand if this is out of scope for Unifont.

Regardless of if and how Unifont uses these glyphs, I intend to make two new fonts to compliment and extend Unifont. One will use the monochrome emoji style glyphs and the other will use the color emoji style glyphs. Both will include full emoji support for sequences and variations. These fonts will only contain emoji glyphs and their text variants. Unifont has full permission to use and redistribute these fonts and glyphs when they become available.
